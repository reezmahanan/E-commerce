# ADR-001: Use Repository Pattern for Data Access

## Status
ACCEPTED

## Context
The application currently has business logic directly accessing the database through raw SQL queries. This creates tight coupling between business logic and the database layer, making testing difficult and reducing maintainability.

## Decision
We will implement the Repository Pattern to abstract all database operations. Each entity will have its own repository class that handles all CRUD operations. Business logic will only interact with repositories, not with the database directly.

## Alternatives Considered
1. **Continue with direct SQL** - Rejected because it creates tight coupling and makes testing difficult.
2. **Use an ORM** - Considered but rejected because it adds complexity and reduces control over queries.
3. **Use Query Builder** - Considered but rejected because it doesn't provide enough abstraction.

## Consequences
- Improved testability through mocking
- Better separation of concerns
- Easier migration between databases
- Additional code overhead
